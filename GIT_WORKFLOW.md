# GIT_WORKFLOW.md

## Назначение

Этот файл определяет единственные допустимые git-сценарии для этого репозитория.

## Базовое правило

- По умолчанию не выполнять git-действия.
- После обычной задачи показывать только изменённые файлы, короткое summary и предлагаемый commit message.
- Не делать commit, push, merge, rebase и переключение веток без явной команды пользователя.

## Политика веток

- `develop` - основная рабочая ветка для повседневной разработки.
- `main` - только для стабильных release-ready изменений.
- `merge main` - отдельное release-действие, а не автоматическое продолжение после `git develop`.

## Запрещено по умолчанию

- `git add .`
- `git commit -a`
- `git push`
- `git push --force`
- `git rebase`
- `git merge`
- переключение веток без явной команды
- коммит `.env`, секретов, токенов, runtime-файлов, кэшей, логов, дампов и backup-файлов

## Разрешённые команды пользователя

Предпочтительные полные команды:

- `сделай git develop по правилам из AGENTS.md`
- `сделай merge main по правилам из AGENTS.md`

Допустимые алиасы:

- `git develop`
- `merge main`

## Сценарий: `git develop`

Если пользователь просит `git develop`, выполнять только этот сценарий:

1. Выполнить `git status`.
2. Убедиться, что текущая ветка - `develop`.
3. Если есть подозрительные посторонние изменения, остановиться и объяснить причину.
4. Показать:
   - какие файлы затронуты;
   - что было изменено;
   - что следующим шагом будет commit и push в `develop`;
   - какой commit message предлагается.
5. Добавить только нужные файлы:
   - `git add <только нужные файлы>`
6. Выполнить commit:
   - `git commit -m "<осмысленное сообщение>"`
7. Выполнить push:
   - `git push origin develop`

### Ограничения

- Не использовать `git add .`, если это может захватить лишние файлы.
- Не коммитить секреты, `.env`, runtime-файлы и временные артефакты.
- Не использовать `--force`.
- Не использовать `rebase`.
- Не делать merge других веток в рамках этого сценария.
- Если есть сомнение, остановиться.

## Сценарий: `merge main`

Если пользователь просит `merge main`, считать это release-действием из `develop` в `main`.

1. Проверить, что `git status` чистый.
2. Если рабочее дерево не чистое, остановиться и показать причину.
3. Убедиться, что текущая ветка - `develop`, либо перейти в неё только если дерево чистое и это безопасно.
4. Показать:
   - что рабочее дерево чистое;
   - какие коммиты есть в `develop` и отсутствуют в `main`;
   - что следующим шагом будет `develop -> main`.
5. Переключиться на `main`:
   - `git checkout main`
6. Подтянуть актуальный `main`:
   - `git pull origin main`
7. Выполнить merge:
   - `git merge develop`
8. Если конфликтов нет, выполнить push:
   - `git push origin main`
9. Вернуться на `develop`:
   - `git checkout develop`

### Ограничения

- Не делать force push.
- Не использовать rebase.
- Не решать merge-конфликты автоматически без явного объяснения.
- Если возник конфликт, остановиться и показать проблемные файлы.
- Если есть сомнение, что в `develop` лежат неготовые изменения, остановиться до merge.

## Формат commit message

Использовать короткие осмысленные сообщения:

- `fix(scope): short description`
- `feat(scope): short description`
- `refactor(scope): short description`
- `chore(scope): short description`
